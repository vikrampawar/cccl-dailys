### Getting Started

| # | Tip | Source |

### Prompting Fundamentals

| # | Tip | Source |
|---|-----|--------|
| 1 | Be specific: instead of "add tests for foo.py" say "write a test for foo.py covering the edge case where the user is logged out. Avoid mocks." Instead of "fix the login bug" say "users report login fails after session timeout. Check auth flow in `src/auth/`, especially token refresh. Write a failing test first, then fix it." | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 2 | Point Claude at existing patterns: "look at how existing widgets are implemented. `HotDogWidget.php` is a good example. Follow the pattern to implement a new calendar widget." | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 3 | Limit scope to roughly "30 minutes worth of human coding" per task. Request stub functions first, then fill each one individually. | [HN community](https://news.ycombinator.com/item?id=44362244) |
| 4 | "Don't accept the first solution. Push Claude to do better — it usually can." | [Boris Cherny — howborisusesclaudecode.com](https://howborisusesclaudecode.com) |
| 5 | Write detailed specs to reduce ambiguity before delegating work to Claude. Documentation-driven development dramatically improves output quality vs vague requests. | [Boris Cherny — 10 Tips](https://paddo.dev/blog/claude-code-team-tips/) |
| 6 | Challenge Claude as a peer: "Grill me on these changes and don't make a PR until I pass your test." Demand proof: "Prove to me this works" with behavior diffs between branches. | [Boris Cherny — howborisusesclaudecode.com](https://howborisusesclaudecode.com) |
| 7 | For non-developers: create custom CLAUDE.md instructions stating you're a designer/non-coder who needs detailed explanations and smaller, incremental changes. | [How Anthropic Teams Use Claude Code](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf) |

### Verification & Safety

| # | Tip | Source |
|---|-----|--------|
| 1 | Always give Claude a way to verify its own work — this is the single highest-leverage thing you can do. Include tests, screenshots, expected outputs, or verification commands. Provides a 2–3× quality improvement. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 2 | Start from a clean git state and commit checkpoints regularly so you can easily revert if Claude goes off track. Use a "try and rollback" methodology. | [How Anthropic Teams Use Claude Code](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf) |
| 3 | After two failed corrections in a session, run `/clear` and start fresh with a better initial prompt. A clean session with a better prompt almost always outperforms a long session with accumulated corrections. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 4 | "Treat it like a slot machine" — commit your state, let Claude work autonomously for 30 minutes, then either accept the result or start fresh. Starting over often has a higher success rate than wrestling with corrections. | [How Anthropic Teams Use Claude Code](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf) |
| 5 | Expect 10–20% of sessions to be abandoned due to unexpected scenarios. That is normal and fine. | [Boris Cherny — InfoQ](https://www.infoq.com/news/2026/01/claude-code-creator-workflow/) |

## Intermediate Tips

### CLAUDE.md Configuration

| # | Tip | Source |
|---|-----|--------|
| 1 | Keep CLAUDE.md concise — under ~500 lines, ideally under 200. For each line, ask: "Would removing this cause Claude to make mistakes?" If not, cut it. Bloated CLAUDE.md files cause Claude to ignore your actual instructions. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 2 | Use emphasis like "IMPORTANT" or "YOU MUST" to improve adherence to critical rules. If Claude ignores a rule, the CLAUDE.md is likely too long and the rule is getting lost. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 3 | CLAUDE.md files support `@path/to/import` syntax to import other files. Example: `@docs/git-instructions.md` or `@~/.claude/my-project-instructions.md`. Supports up to 5 levels of recursive imports. | [Memory](https://code.claude.com/docs/en/memory) |
| 4 | Place CLAUDE.md files hierarchically: `~/.claude/CLAUDE.md` (global), project root (team, check into git), parent dirs (monorepo), child dirs (loaded on demand when working in that directory). | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 5 | Maintain a single CLAUDE.md checked into git that the whole team contributes to. Anytime Claude does something incorrectly, add it so Claude knows not to repeat it. This is "Compounding Engineering." | [Boris Cherny — howborisusesclaudecode.com](https://howborisusesclaudecode.com) |
| 6 | Structure CLAUDE.md around WHAT (tech stack, codebase structure), WHY (project purpose), and HOW (build tools, test execution, verification). | [HumanLayer](https://www.humanlayer.dev/blog/writing-a-good-claude-md) |
| 7 | Never auto-generate your CLAUDE.md blindly. It is the highest-leverage configuration point. Run `/init` for a starter, then heavily refine. Start with an empty file and only add instructions when you find yourself repeating corrections. | [HumanLayer](https://www.humanlayer.dev/blog/writing-a-good-claude-md) |
| 8 | Use `.claude/rules/` for path-specific instructions. Add YAML frontmatter with `paths: ["src/api/**/*.ts"]` to only load rules when Claude works with matching files. Saves context vs loading everything in CLAUDE.md. | [Memory](https://code.claude.com/docs/en/memory) |
| 9 | At end of sessions, ask Claude to summarize completed work and suggest improvements to CLAUDE.md. This creates a continuous improvement loop where documentation gets refined based on actual usage. | [How Anthropic Teams Use Claude Code](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf) |
| 10 | Add specific tool-calling corrections to CLAUDE.md to prevent repeated mistakes, such as "run pytest not run and don't cd unnecessarily — just use the right path." | [How Anthropic Teams Use Claude Code](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf) |
| 11 | CLAUDE.md fully survives compaction — after `/compact`, Claude re-reads it from disk. If an instruction disappeared after compaction, it was given only in conversation, not written to CLAUDE.md. | [Memory](https://code.claude.com/docs/en/memory) |

### Context Management

| # | Tip | Source |
|---|-----|--------|
| 1 | Run `/clear` between unrelated tasks to reset context. Context is the most important resource — LLM performance degrades as the context window fills. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 2 | Use `/compact <instructions>` to customize what gets preserved during summarization, e.g. `/compact Focus on the API changes`. Add permanent compaction rules to CLAUDE.md. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 3 | Use `/btw` for quick side questions that don't enter conversation history — keeps context clean. The answer appears in a dismissible overlay. | [Interactive Mode](https://code.claude.com/docs/en/interactive-mode) |
| 4 | Use `Esc + Esc` or `/rewind` to open the rewind menu. Select a checkpoint and choose "Summarize from here" to condense messages while keeping earlier context intact. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 5 | Track context usage continuously by configuring a custom status line with `/statusline`. Run `/cost` to check token usage, or `/context` to see what is consuming context space (including MCP overhead). | [Manage Costs](https://code.claude.com/docs/en/costs) |
| 6 | Use subagents for investigation to keep your main context clean: "use subagents to investigate how our auth system handles token refresh". The subagent explores in a separate context window and reports back a summary. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 7 | Run `/insights` once a week. It analyzes recent sessions and surfaces patterns — what's working, what's failing, where you're spending time. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |

### Plan Mode & Workflow

| # | Tip | Source |
|---|-----|--------|
| 1 | Follow the 4-phase workflow: **Explore** (Plan Mode, read-only) → **Plan** (create implementation plan, edit with `Ctrl+G`) → **Implement** (Normal Mode) → **Commit**. Skip planning for simple changes. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 2 | Start every complex task in Plan mode (`Shift+Tab` twice). Go back and forth with Claude until you like the plan, then switch to auto-accept for implementation. "A good plan is really important!" | [Boris Cherny — howborisusesclaudecode.com](https://howborisusesclaudecode.com) |
| 3 | Have Claude interview you before building a feature: "I want to build [brief description]. Interview me in detail using the AskUserQuestion tool." Then write the spec and start a fresh session to implement. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 4 | Develop task classification intuition: peripheral features and prototyping work well in auto-accept mode (async), while core business logic and critical fixes need synchronous supervision with detailed prompts. | [How Anthropic Teams Use Claude Code](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf) |
| 5 | Tell Claude to "commit your work as you go" and let it work autonomously with periodic check-ins, rather than asking targeted questions for code snippets. Results in more comprehensive solutions. | [How Anthropic Teams Use Claude Code](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf) |
| 6 | When a solution feels mediocre, restart: "Knowing everything you know now, scrap this and implement the elegant solution." | [Boris Cherny — howborisusesclaudecode.com](https://howborisusesclaudecode.com) |

### CLI & Automation

| # | Tip | Source |
|---|-----|--------|
| 1 | Use `claude -p "prompt"` for non-interactive (headless) mode in scripts, CI, and pipelines. Add `--output-format json` or `--output-format stream-json` for structured output. | [CLI Reference](https://code.claude.com/docs/en/cli-reference) |
| 2 | Pipe data through Claude: `cat build-error.txt \| claude -p 'explain the root cause' > output.txt`. Chain with Unix tools: `git diff main --name-only \| claude -p "review these changed files for security issues"`. | [Common Workflows](https://code.claude.com/docs/en/common-workflows) |
| 3 | Use `--max-turns 3` to limit agentic turns and `--max-budget-usd 5.00` to cap spending in headless mode. | [CLI Reference](https://code.claude.com/docs/en/cli-reference) |
| 4 | Use `claude -p` as a custom linter in package.json: `"lint:claude": "claude -p 'look at changes vs. main and report typos'"`. | [Common Workflows](https://code.claude.com/docs/en/common-workflows) |
| 5 | Use `--allowedTools` to scope permissions for batch operations: `claude -p "fix linting" --allowedTools "Edit,Bash(npm run lint)"`. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 6 | Set up terminal aliases: `alias c='claude'`, `alias ch='claude --chrome'`. Use flags: `c -c` (continue), `c -r` (resume). | [YK — claude-code-tips](https://github.com/ykdojo/claude-code-tips) |
| 7 | Increase timeouts for long-running operations with `MCP_TOOL_TIMEOUT` and `BASH_MAX_TIMEOUT_MS` environment variables. | [Shrivu Shankar](https://blog.sshh.io/p/how-i-use-every-claude-code-feature) |

### Model Selection & Costs

| # | Tip | Source |
|---|-----|--------|
| 1 | Use Sonnet for most tasks, reserve Opus for complex architectural decisions. Switch with `/model` mid-session. For simple subagent tasks, specify `model: haiku`. | [Manage Costs](https://code.claude.com/docs/en/costs) |
| 2 | Use `opusplan` model alias: Opus for planning/reasoning, Sonnet for code execution. Set with `/model opusplan`. | [Model Config](https://code.claude.com/docs/en/model-config) |
| 3 | Use 1M context window with `opus[1m]` or `sonnet[1m]` model aliases for long sessions with large codebases. | [Model Config](https://code.claude.com/docs/en/model-config) |
| 4 | Adjust effort with `/effort low`, `/effort medium`, `/effort high`, or `/effort max` (Opus only). Lower effort is faster and cheaper for simple tasks. | [Manage Costs](https://code.claude.com/docs/en/costs) |
| 5 | Prefer CLI tools (`gh`, `aws`, `gcloud`) over MCP servers — they are more context-efficient because they don't add persistent tool definitions. Disable unused MCP servers via `/mcp`. | [Manage Costs](https://code.claude.com/docs/en/costs) |
| 6 | Install code intelligence plugins for typed languages — a single "go to definition" call replaces grep + reading multiple candidate files. Run `/plugin` to browse the marketplace. | [Manage Costs](https://code.claude.com/docs/en/costs) |
| 7 | Average Claude Code cost is ~$6/developer/day with 90% of users under $12/day. Use `/cost` to track session costs. Use `/stats` for subscription usage patterns. | [Manage Costs](https://code.claude.com/docs/en/costs) |

### Hooks

| # | Tip | Source |
|---|-----|--------|
| 1 | Use hooks for actions that must happen every time with zero exceptions (unlike CLAUDE.md which is advisory). Ask Claude to write hooks: "Write a hook that runs eslint after every file edit." Configure in `.claude/settings.json` and browse with `/hooks`. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 2 | Auto-format after edits with a PostToolUse hook: `"PostToolUse": [{"matcher": "Write\|Edit", "hooks": [{"type": "command", "command": "prettier --write $CLAUDE_FILE_PATH"}]}]` | [Boris Cherny — howborisusesclaudecode.com](https://howborisusesclaudecode.com) |

| 4 | Block dangerous commands with a PreToolUse hook. Create a script that reads JSON from stdin, checks `tool_input.command` with jq, and exits with code 2 to block. | [Hooks](https://code.claude.com/docs/en/hooks) |
| 5 | View all configured hooks by typing `/hooks` in Claude Code. Shows a read-only browser with event counts, matcher details, and source locations. | [Hooks](https://code.claude.com/docs/en/hooks) |

### Security & Permissions

| # | Tip | Source |
|---|-----|--------|
| 1 | Use `/permissions` to allowlist safe commands (like `npm run lint`, `git commit`) to reduce permission prompt interruptions. Use wildcard syntax: `"Bash(bun run *)"`, `"Edit(/docs/**)"`. | [Boris Cherny — howborisusesclaudecode.com](https://howborisusesclaudecode.com) |
| 2 | Use `--dangerously-skip-permissions` only in sandboxed environments without internet. Use `/sandbox` for OS-level file and network isolation. | [Anthropic Best Practices](https://code.claude.com/docs/en/best-practices) |
| 3 | Block credential reads via permission deny rules in settings.json: `~/.ssh/**`, `~/.aws/**`, `~/.kube/**`, `~/.npmrc`, `~/.pypirc`. | [Trail of Bits](https://github.com/trailofbits/claude-code-config) |
| 4 | Set `enableAllProjectMcpServers: false` (default) since `.mcp.json` files live in git — prevents supply-chain injection via malicious MCP configs in cloned repos. | [Trail of Bits](https://github.com/trailofbits/claude-code-config) |
| 5 | Use `"dontAsk"` permission mode to auto-deny tools unless pre-approved. Useful for CI/CD: `claude -p "..." --permission-mode dontAsk --allowedTools "Read,Grep,Glob"`. | [Permissions](https://code.claude.com/docs/en/permissions) |
| 6 | Permission rules use deny → ask → allow precedence. Deny always wins. The space before `*` matters: `Bash(ls *)` matches `ls -la` but not `lsof`. | [Permissions](https://code.claude.com/docs/en/permissions) |
